# Async Runtime

### Общая идея и мотивация

Современный мир программного обеспечения тяжело представить без сетевого взаимодействия.
Предположим, у нас есть простая программа сервера, которая хотела бы принимать клиентов, используя [сокеты](https://en.wikipedia.org/wiki/Berkeley_sockets):

``` cpp
// Creating socket...

::listen(server_fd, 10);

int conn_fd = ::accept(server_fd, nullptr, nullptr);

// Handling connection...
```

Возникает вопрос, как с таким подходом обслуживать одновременно несколько клиентов.
Можно попробовать выделять на каждого нового клиента новый поток исполнения (например, с помощью `std::thread`), и такой подход действительно будет работать некоторое время &mdash; примерно до [10K](https://en.wikipedia.org/wiki/C10k_problem) одновременных клиентов.
Но что делать, если необходимо обрабатывать сотни тысяч, миллионы запросов?

Решением проблемы выступает взятие некоторых обязанностей ядра операционной системы на себя.
Так появляются [корутины](https://en.wikipedia.org/wiki/Coroutine) (сюда же можно отнести [зелёные потоки](https://en.wikipedia.org/wiki/Green_thread) и [файберы](https://en.wikipedia.org/wiki/Fiber_(computer_science))).
За планирование и исполнение корутин должен отвечать некоторый рантайм, который может быть фреймворком для выбранного языка ([std::execution](https://wg21.link/P2300), [Tokio](https://tokio.rs/), [Userver](https://userver.tech/), [Seastar](https://seastar.io/)), или же полностью быть в него интегрирован, как, например, в [Go](https://pkg.go.dev/runtime).

Так или иначе, выбор рантайма является ключевым при написании дальнейшего программного продукта.

Как и с языками программирования, среди подобных решений возникает несколько моделей работы с асинхронностью, из которых интересны две:

1. *Функциональная*, где асинхронные функции можно рассматривать как функции, возвращающие [future](https://en.wikipedia.org/wiki/Futures_and_promises) &mdash; некоторое асинхронное вычисление, которое вернёт результат не сразу же после завершения функции.
Так, например, устроены асинхронные функции в [Rust](https://doc.rust-lang.org/book/ch17-00-async-await.html) или в [Seastar](https://seastar.io/futures-promises/).

2. *Императивная*, где асинхронные функции не отличаются от обычных, однако с помощью языковых средств могут быть запущены в новом потоке исполнения.
Что делать с результатом вычисления &mdash; остается на откуп программиста.
Именно такой подход заложен в парадигму Go в виде [горутин](https://go.dev/tour/concurrency/1).

В этом задании вам предлагается последовать второй парадигме и написать свой асинхронный рантайм с помощью возможностей C++.

### Декомпозиция

Предлагается декомпозировать решение на три слабо связанных компонента:

- `coro` &mdash; интерфейс взаимодействия пользователя с асинхронными вычислениями, выраженными через С++20 корутины;
- `sched` &mdash; логика планирования и исполнения произвольных задач;
- `sync` &mdash; примитивы синхронизации, необходимые для написания конкурентного кода.

## Scheduler

Единицей исполнения в фреймворке предлагается выделить некоторое семейство типов `sched::Resumable<Scheduler>`, имеющих единственный виртуальный метод `void resume(Scheduler&)` &mdash; часть некоторого вычисления, которая должна быть исполнена на переданном планировщике. 
Можно заметить, что такая реализация является достаточно абстрактной и не завязывается на конкретный механизм исполнения, будь то корутины C++20, [Boost.Fiber](https://www.boost.org/doc/libs/latest/libs/fiber/doc/html/index.html), или даже синхронная реализация на `std::function`.

В качестве шаблонного параметра `Scheduler` задаётся базовый тип для некоторого семейства планировщиков, на которых задача сможет исполняться. При этом все планировщики в этой иерархии реализуют интерфейс, состоящий из метода `spawn`, который принимает `sched::Resumable<Scheduler>&`, где `Scheduler` &mdash; корень иерархии планировщиков.

Технически, параметризацию вычисления планировщиком можно в некотором смысле рассматривать как тэггирование: благодаря этому библиотека может иметь наследников `Resumable`, поддерживающих разные семейства планировщиков.

Однако, с такими абстракциями ничего не мешает сделать и resumable, поддерживающий сразу несколько планировщиков &mdash; всего лишь нужно унаследоваться от нескольких `Resumable`, параметризованных корнями разных иерархий планировщиков.

Такая декомпозиция позволяет написать достаточно абстрактные планировщики, которые легко переиспользовать для различных случаев.

### Интрузивность

Чтобы добиться наиболее эффективной реализации с минимумом аллокаций памяти, в качестве хранилища для исполняемых задач планировщики обычно используют интрузивные структуры данных (например, вот что написано в [std::execution](https://wg21.link/P2300R7#spec-execution.contexts.run_loop:~:text=%5BNote%3A%20Implementations%20are%20encouraged%20to%20use%20an%20intrusive%20queue%20of%20operation%20states%20to%20hold%20the%20work%20units%20to%20make%20scheduling%20allocation%2Dfree.%20%E2%80%94%20end%20note%5D)).

Самым типичным случаем является использование интрузивных списков, и в этом задании предлагается поддержать лишь планировщики, использующие их. Таким образом, ваши планировщики будут наследоваться от `sched::IntrusiveListScheduler` и реализовывать `spawn(Resumable<IntrusiveListScheduler>&)`.

Это, разумеется, значит, что `Resumable<IntrusiveListScheduler>` должен предусматривать интрузивное API.

Иногда возможно хочется иметь владеющие планировщики, или планировщики на других интрузивных структурах (например, [в ядре Linux](https://developer.ibm.com/tutorials/l-completely-fair-scheduler/) используются интрузивные деревья).
Сам интерфейс `sched::Resumable` поддерживает такую возможность, однако их реализация выходит за рамки этого задания.

### Планировщик `RunLoop`

Первым, и самым простым из планировщиков, требуемых в задании, является `sched::RunLoop` &mdash; однопоточный планировщик.

Требования и инварианты методов описаны в `sched/run-loop.h`.
Метод `run` блокирует текущий поток исполнения до тех пор, пока не выполнит все поставленные задачи.

Подобный планировщик, из-за его детерминированности, очень удобно использовать для тестирования кода. Он также подходит для исполнения IO-bound задач.

### Планировщик `ThreadPool`

Другим планировщиком выступает `sched::ThreadPool`, порождающий ровно столько потоков, сколько было задано в конструкторе.
Все запущенные потоки должны быть остановлены в деструкторе.

Подобный планировщик можно использовать для простых compute-bound задач, однако он не выдерживает высоких нагрузок (см. ниже пункт *Дальнейшие улучшения*).

## Coroutines

В общем смысле корутина &mdash; это функция, которая может быть остановлена и возобновлена.
Такое определение не накладывает на реализацию никаких дополнительных ограничений, однако в теории сформировались несколько подходов:

- *Stackful* &mdash; технически корутина представляет собой облегчённый поток ОС и управляет сохранением состояния при остановке/возобновлении с помощью выделения собственного стека. Такой подход, например, принят в [Userver](https://userver.tech/d7/dc8/md_en_2userver_2stack.html) или в [folly](https://github.com/facebook/folly/tree/main/folly/fibers).

- *Stackless* &mdash; в этом случае корутина является отдельной сущностью, которую компилятор разворачивает в некоторый автомат, состояниями которого выступает *стековый фрейм* текущей асинхронной функции. Такой подход был принят в [C++20 Coroutines](https://en.cppreference.com/w/cpp/language/coroutines.html) и активно развивается.

Каждый из этих подходов имеет свои плюсы и минусы (которые вы наверняка обсуждали на лекциях), однако в данной задаче мы остановимся на втором из них.

### `coro::Coroutine`

Асинхронной функцией мы будем считать функцию, возвращающую объект типа `coro::Coroutine`, и содержащую ключевые слова `co_await`/`co_return`.
Подобно Go, мы не будем позволять таким функциям возвращать значения, средства для этого будут обсуждены позже.

Так как в нашей модели не имеется естественной точки, в которой значение асинхронной функции должно быть получено, а равно и не имеется естественной точки для проброса исключения, с исключением, которое вылетает из тела корутины, нельзя сделать ничего сильно лучше, чем вызвать `std::terminate` (аналогично тому, что происходит [при вылетании исключения из функции потока](https://timsong-cpp.github.io/cppwp/n4950/except.terminate#1.10)).

Заметим, что `coro::Coroutine` предполагается  объектом, не владеющим фреймом корутины. Над тем, какое время жизни имеет сам фрейм, и кто должен управлять его временем жизни, предлагается как следует осмыслить самостоятельно.

### `coro::go`

Интерфейсом для запуска корутин в фреймворке выступает функция `go`:

```cpp
coro::go(pool, [] -> coro::Coroutine {
    std::cout << "Hello world!" << std::endl;

    coro::go([] -> coro::Coroutine {
        std::cout << "Inner one!" << std::endl;
        co_return;
    });

    co_return;
});
```

С точки зрения нашего фреймворка асинхронная функция будет представлять собой серию вызовов `resume` у `sched::Resumable`, исполненных на планировщике, переданном в `coro::go`.

Удобно иметь две перегрузки:
- Корневую &mdash; принимающую планировщик и функциональный объект, порождающий объект `coro::Coroutine`;
- Дочернюю &mdash; принимающую только функциональный объект и наследующую планировщик от родительской корутины.


### `coro::yield`

В отличие от Go, в вашем фреймворке предлагается поддержать модель [кооперативной многозадачности](https://en.wikipedia.org/wiki/Cooperative_multitasking), то есть исполняемые задачи не будут прерываться планировщиком, а будут добровольно отдавать поток исполнения.

Делать это предлагается с помощью функции `coro::yield`:

```cpp
co_await coro::yield();
```

Она должна передать поток исполнения обратно текущему планировщику.

## Synchronization

Для [корректной](https://timsong-cpp.github.io/cppwp/n4950/intro.multithread#def:data_race) работы программы при наличии конкурентных операций необходимо добиваться синхронизации.

Пользоваться стандартными примитивами типа `std::mutex` в теле корутины категорически запрещено: подобное может повлечь блокировку целого исполнителя, что чревато возникновением [дедлока](https://en.wikipedia.org/wiki/Deadlock_(computer_science)).

Поэтому в задании требуется реализовать собственные примитивы синхронизации, которые могут снимать с исполнения корутины.

Общим требованием к примитивам синхронизации, которое вытекает из устройства планировщиков, является отсутствие аллокаций.

### Event

`sync::Event` &mdash; это объект, который можно дождаться с помощью метода `wait` и пробудить всех ожидающих исполнителей с помощью метода `emit`.

```cpp
sync::Event done;

auto producer = [&] {
    co_await coro::yield();
    done.emit();
};

coro::go(producer);
co_await event.wait();
```

В задании требуется реализовать лишь oneshot Event, то есть такой, который можно активировать лишь единожды.

### Mutex

`sync::Mutex` &mdash; примитив, позволяющий программе достигать **mutual exclusion** &mdash; свойства, гарантирующего, что лишь один поток исполнения имеет доступ к определенному участку кода (критической секции).

Вопрос интерфейса для подобного примитива является дискуссионным.
С одной стороны, в [production фреймворке](https://github.com/facebook/folly/blob/main/folly/synchronization/DistributedMutex-inl.h#L462-L464) хотелось бы достаточно эффективную реализацию (см. ниже пункт *Flat combining*), и тогда выбор падает на интерфейс вида:

```cpp
mutex.run([] {
    // critical section
});
```

С другой, хотелось бы иметь нативный для языка интерфейс и пользоваться идиомой RAII:

```cpp
{
    auto guard = co_await mutex.lock();
    // critical section
}
```

Однако, в последнем случае мы несколько ограничены возможностями языка в связи с отсутствием [красных](https://journal.stuffwithstuff.com/2015/02/01/what-color-is-your-function/) [деструкторов](https://wg21.link/P2849).


В задании предлагается следовать третьему варианту интерфейса, который менее всего ограничивает свободу реализации:

```cpp
co_await mutex.lock();
// critical section
co_await mutex.unlock();
```

#### Flat combining

Можно заметить, что, как примитив, `Mutex` дает в программе естественный *bottleneck*: критические секции для сохранения корректности не могут исполняться "лучше", чем последовательно.
Однако, исполняя критическую секцию попеременно на разных ядрах, мы невольно тратим ресурс на работу протокола когерентности кэшей.

Таким образом, довольно естественно возникает идея [flat combining](https://www.cs.tau.ac.il//research/moran.tzafrir/papers/spaa2010_flat_combining.pdf): давайте не тянуть данные к ядрам, а выполнять секции на одном ядре.

В задании не требуется, но эффективная реализация `sync::Mutex` предполагает использование этой техники.

### Каналы

Естественным остается желание получать из асинхронных функций результаты работы.
В языке Go невозможность горутины вернуть результат компенсируется парадигмой [Share Memory By Communicating](https://go.dev/blog/codelab-share).

В этом задании также предлагается реализовать примитив `sync::BufferedChannel<T>` &mdash; буферизованного канала, подобного [Go-шному](https://go.dev/tour/concurrency/3).

```cpp
auto ch = sync::BufferedChannel<int>(2);

co_await ch.send(1);
co_await ch.send(2);

std::cout << co_await ch.recv() << std::endl;
std::cout << co_await ch.recv() << std::endl;
```

Для простоты, в отличие от Go, предлагается опустить метод `close`.

Подумайте, чем будет в таком случае являться [небуферизованный канал](https://go.dev/tour/concurrency/2) из Go.

## Бонусная версия (lock-freedom)

В базовой версии задания не требуется гарантировать [lock-freedom](https://en.wikipedia.org/wiki/Non-blocking_algorithm#Lock-freedom) для примитивов `sync::Event` и `sync::Mutex`, однако эффективная их реализация достигает этой гарантии.

При желании вы можете попробовать поддержать для них эту гарантию и получить бонусные баллы.

От канала не требуется гарантировать *lock-freedom* (даже в бонусной версии), также как это не гарантируется и в Go.
Однако же подобные реализации существуют, например, в [Kotlin](https://nikitakoval.org/publications/ppopp19-channels.pdf).
Подумайте, какие проблемы могут возникать с реализацией на языке C++, в отличие от Kotlin, в связи с отсутствием сборщика мусора.

## Дальнейшие улучшения

Предложенная декомпозиция позволит независимо расширять возможности получившегося рантайма, так можно:

- Отойти от модели языка Go с detached задачами в сторону [Structured Concurrency](https://www.youtube.com/watch?v=Mj5P47F6nJg);
- Добавить планировщики поддерживающие IO, таймеры или написать хороший, [масштабируемый планировщик](https://www.youtube.com/watch?v=-K11rY57K7k);
- Расширить возможности библиотеки синхронизации, добавив новых примитивов: [RwLock](https://docs.rs/tokio/latest/tokio/sync/struct.RwLock.html), [WaitGroup](https://go.dev/src/sync/waitgroup.go).
